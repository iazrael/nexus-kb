## 概述

用于在执行其他 nexus 命令后，反思本次对话的效率和正确性问题，检测常见反模式，生成针对性的命令优化建议。

**核心目标**：
- 检测工具选择问题（是否使用合适的工具）
- 检测效率问题（冗余读取、步骤顺序）
- 检测格式规范问题（frontmatter、命名、链接）
- 检测命令正确性问题（步骤遗漏、顺序错误）
- 捕获用户纠正反馈并转化为命令改进

---

## 步骤 1：多维度检测规则

分析对话中的决策和执行，检测五类问题：

### 1.1 工具选择问题

| 反模式 | 检测规则 | 建议修复 |
|--------|----------|----------|
| 搜索文件 | 用 `find` 或 `ls` 搜索 | 改用 Glob 工具 |
| 搜索内容 | 用 `grep` 或 `rg` 搜索 | 改用 Grep 工具 |
| 读取文件 | 用 `cat` 或 `head` 读取 | 改用 Read 工具 |
| 编辑文件 | 用 `sed` 或 `awk` 编辑 | 改用 Edit 工具 |
| 过度 Bash | 可用专用工具但用了 Bash | 检查是否有更合适的工具 |

### 1.2 效率问题

| 反模式 | 检测规则 | 建议修复 |
|--------|----------|----------|
| 冗余读取 | Read 全文但只用 frontmatter | 改用 `limit: 20` |
| 批量判断状态 | 对多文件 Read 全文仅判断 status | 先 Glob + 读 frontmatter |
| 重复读取 | 同一文件被 Read 多次 | 缓存结果 |
| 步骤顺序不当 | 先深入后判断 | 加入预扫描阶段 |

### 1.3 格式规范问题

| 反模式 | 检测规则 | 建议修复 |
|--------|----------|----------|
| frontmatter 缺失 | wiki 文件无 YAML frontmatter | 在命令模板中强制要求 |
| frontmatter 字段不规范 | 字段名不符合 snake_case 或格式错误 | 在命令中加入格式检查步骤 |
| 文件命名不规范 | 不符合命名约定（如概念文件应为 PascalCase） | 在命令中加入命名验证 |
| 链接格式错误 | 使用反引号或 HTML 标签而非标准 Markdown 链接 | 在命令中要求 `[文本](路径)` 格式 |

### 1.4 命令正确性问题

| 反模式 | 检测规则 | 建议修复 |
|--------|----------|----------|
| 步骤遗漏 | 跳过命令定义中的必要步骤 | 补充缺失步骤到命令模板 |
| 步骤顺序错误 | 执行顺序与命令定义不一致 | 调整命令模板中的步骤顺序 |
| 输出格式不符 | 实际输出与命令定义的格式模板不一致 | 统一输出格式模板 |
| 索引未更新 | 创建新条目后未更新 index.md | 强制要求索引更新步骤 |

### 1.5 用户纠正反馈

| 检测项 | 说明 | 处理方式 |
|--------|------|----------|
| 用户纠正行为 | 用户说"不对"、"应该这样"等纠正语句 | 提取纠正内容，更新命令模板 |
| 用户补充要求 | 用户说"还应该加 X"、"漏了 Y" | 将补充内容加入命令模板 |
| 用户偏好调整 | 用户说"我更喜欢 X 方式" | 记录到项目记忆作为偏好 |
| 用户否定建议 | 用户说"不需要 X"、"太复杂了" | 移除或简化命令中的相关步骤 |

## 步骤 2：收集对话上下文

1. 回顾本次对话中所有工具调用序列（按时间顺序）
2. 标记关键决策点：工具选择、读取范围、步骤跳转
3. 捕获用户反馈语句：纠正、补充、否定、偏好表达

---

## 步骤 3：逐维度检测

按以下顺序执行检测：

1. **工具选择问题** → 优先检测（影响最大）
2. **效率问题** → 检测 token 消耗
3. **格式规范问题** → 检查输出文件
4. **命令正确性问题** → 对照命令定义
5. **用户纠正反馈** → 分析用户语句

每个维度检测结果记录：
- 问题类型
- 具体位置（工具调用序号或对话位置）
- 受影响的命令文件
- 严重程度（高/中/低）

---

## 步骤 4：生成针对性建议

对每个检测到的问题：

1. 定位需要修改的命令文件路径
2. 给出具体修改内容（新增/修改/删除哪个章节或步骤）
3. 给出预期效果描述

---

## 步骤 5：更新记忆系统

## 注意事项

- **不自动修改命令文件**：报告生成后需用户确认才执行修改
- **检测依赖对话记录**：需要完整保留本次对话的工具调用序列
- **严重程度判断**：
  - 高：影响核心功能正确性或造成显著效率损失
  - 中：影响输出质量或中等效率损失
  - 低：影响代码风格或轻微效率损失
- **用户反馈捕获**：重点关注否定、纠正类语句，偏好类需谨慎判断是否适用全局
- **与 project memory 配合**：每次 evolve 结果写入记忆，形成持续积累
- **定期执行**：建议每完成一批 nexus 命令后执行一次 evolve
